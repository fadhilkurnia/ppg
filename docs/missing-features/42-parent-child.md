# 42 — Parent (Ortu) ↔ Child (Murid)

## TL;DR

Add an `ortu_murid` association linking a parent `users` row to one or more murid `users` rows. Grant the parent read-only access to each linked child's kelas, sesi, raport, and notifications via scope-aware filtering. Surface a "My children" dashboard on the SPA.

## 1. Data model

DDL is in [10-domain-model-evolution.md §3.19](./10-domain-model-evolution.md). Key shape:

```sql
CREATE TABLE ortu_murid (
    ortu_user_id   TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    murid_user_id  TEXT NOT NULL REFERENCES users(id) ON DELETE RESTRICT,
    relation       TEXT NOT NULL DEFAULT 'parent' CHECK (relation IN ('parent','guardian')),
    is_primary     INTEGER NOT NULL DEFAULT 0 CHECK (is_primary IN (0,1)),
    created_at     TEXT NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    PRIMARY KEY (ortu_user_id, murid_user_id)
);

CREATE INDEX idx_ortu_murid_murid ON ortu_murid(murid_user_id);
```

Notes:

- A child may have multiple parents/guardians (split family, extended family).
- A parent may have multiple children.
- `is_primary` marks the "official" parent for raport delivery.

## 3. Permission helper

Extend the existing claims helper:

```go
type ScopedClaims struct {
    *Claims
    PrimaryScopeID    string
    EffectiveScopeIDs map[string]struct{}
    LinkedMuridIDs    map[string]struct{} // populated for ortu role
    IsAdmin           bool
}

func (c *ScopedClaims) CanReadUser(targetID string) bool {
    if c.IsAdmin { return true }
    if c.UserID == targetID { return true }
    if _, ok := c.LinkedMuridIDs[targetID]; ok { return true }
    // ... scope check ...
    return false
}
```

Apply this check uniformly across user-detail endpoints, raport endpoints, sesi participation, etc. The result: an ortu can read everything *about* their child but cannot mutate.

## 4. Notification fan-out

When `kelas.enrolled`, `materi.graded`, `sesi.started`, etc. produce a notification for a murid, the notifier also looks up `ortu_murid` and sends a parallel notification to each linked ortu. The notifier orchestrates this:

```go
recipients := []string{muridID}
recipients = append(recipients, ortuOf(muridID)...)
notifier.Send(ctx, Event{ UserIDs: recipients, ... })
```

Mark notifications targeted at ortu with a `meta.role = "ortu"` so the SPA can format the body differently ("Anak Anda, X, ...").

## 5. SPA routes

- `/_authed/anak` — parent dashboard: card per child with attendance, latest sesi, last grade.
- `/_authed/anak/{muridId}` — child detail: same shape as `/me` for that child.
- `/_authed/anak/{muridId}/raport` — child's raport.
- `/_authed/anak/{muridId}/sesi` — child's sesi list.

Persona check in the layout: a user with role `ortu` and ≥ 1 linked child sees the **Anak** tab in the sidebar.

## 7. Open questions

- **Self-claim by code**: an ortu signs up and links to a child via a one-time invite code generated by the kelas's guru. Defer; useful but adds an invite table.
- **Multi-ortu read consistency**: if two ortu of one murid log in simultaneously and modify their own profile, are there conflicts? No — they edit different `users` rows.
- **Ortu push to murid**: ortu request a session note from guru. Defer; rely on out-of-band messaging until needed.
